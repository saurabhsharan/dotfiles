# xremap Python DSL Configuration

This directory contains a custom, declarative Python DSL (Domain Specific Language) for managing your `xremap` configuration. 

Managing raw YAML for `xremap` can quickly become tedious due to duplication, especially when you want to share sets of keybindings (like Emacs navigation) across multiple applications or define hardware remappings for specific devices.

This framework solves that by allowing you to define your configuration in Python using classes, variables, and dictionary merging, which then compiles down to standard `config.yml`.

## Features

- **Declarative Python Syntax:** Write your config using simple Python classes.
- **Inheritance & Merging:** Share snippets of keybindings using standard Python dictionary merging (`|`). Define base classes (like `BaseBrowser`) and inherit from them to avoid repetition.
- **List Variables:** Group applications into lists (e.g., `TERMINALS`, `BROWSERS`) and use list concatenation to easily define `only` or `not` scopes.
- **Conflict Detection:** The compiler automatically detects and warns you if you define the same shortcut multiple times for the same application. It distinguishes between actual conflicts (different actions) and redundant duplicates (same action).
- **HTML Documentation Generation:** Every time you compile, an aesthetic, interactive `shortcuts.html` is generated. It allows you to browse your shortcuts by application, sorted by specificity.
- **Standardized Casing:** The Python DSL encourages a standard casing convention (e.g., `Super-Shift-left`) and normalizes shortcuts under the hood for accurate conflict detection.

## Files

- `config.py`: **Your main configuration file.** This is where you define your variables, snippets, modmaps, and keymaps. Edit this file to change your shortcuts.
- `xremap_dsl.py`: The engine behind the scenes. It provides the base classes (`Modmap`, `Keymap`, `App`, `Device`), handles the YAML generation, checks for conflicts, and builds the HTML documentation.
- `config.yml`: The generated file that `xremap` actually reads. **Do not edit this file directly.**
- `shortcuts.html`: The generated interactive documentation.

## Usage

1. Open `config.py` in your favorite editor.
2. Add or modify your shortcuts using the class-based syntax.
3. Compile the configuration by running:

```bash
python3 config.py
```

This will:
1. Validate your shortcuts and print any conflicts or duplicates to the console.
2. Generate the `config.yml` file.
3. Generate the `shortcuts.html` documentation.

## DSL Design: Constants vs. Inheritance

The DSL provides two powerful ways to share configuration, but they serve different architectural purposes: **Composition** vs. **Hierarchy**.

### 1. Constants (Dictionaries) are for *Composition* (Mix-ins)
Constants like `EMACS_NAV` and `MAC_ARROWS` act as modular building blocks. You use them when a set of shortcuts applies to multiple, **completely unrelated applications** that do not share a common lineage.
* **Example:** You want Emacs-style text navigation (`Ctrl-a: home`, `Ctrl-e: end`) in Chrome (a browser), Roam PWA (a web app), and potentially a text editor.
* By defining `EMACS_NAV` as a dictionary, you can easily "mix it in" to any class using the `|` merge operator, regardless of what that class represents.

### 2. Inheritance (Base Classes) is for *Hierarchy* (Specialization)
Inheritance like `BaseBrowser` is used when applications share a **core identity** and you need to define a strict baseline that can be incrementally overridden.
* **Example:** Firefox and Chrome are both web browsers. They both need tab management (`Super-t`), zooming (`Super-minus`), and tab switching.
* Instead of copying all browser shortcuts into both a `Firefox` class and a `Chrome` class, you put them in a `BaseBrowser` base class.
* Inheritance shines because it handles **overrides** perfectly. `BaseBrowser` defines `Super-w: Ctrl-w`. Firefox inherits it perfectly. Chrome inherits it, but *overrides* it specifically with `Super-w: Ctrl-f4`.

**Rule of Thumb:**
* Use **Dictionaries** when you have a modular set of keys you want to sprinkle across wildly different apps.
* Use **Inheritance** when you have a family of apps that should behave identically 90% of the time, but need specific exceptions or additions.

## Syntax Examples

### Variables & Snippets

Define lists of applications and dictionaries of shared shortcuts:

```python
TERMINALS = ["org.wezfurlong.wezterm", "kitty"]
EMACS_NAV = {"Ctrl-a": "home", "Ctrl-e": "end"}
```

### Modmaps

Change hardware keys:

```python
class CapsLockToCtrl(Modmap):
    name = "Change Caps Lock to Control"
    remap = {"CapsLock": "Ctrl_L"}
```

### Keymaps & Inheritance

Define specific application shortcuts and inherit shared logic:

```python
class BaseBrowser(Keymap):
    is_base = True # Tells the compiler this is a template, not a final keymap
    exact_match = True
    remap = {
        "Super-l": "Ctrl-l",
        "Super-t": "Ctrl-t"
    }

class Firefox(BaseBrowser):
    name = "Firefox"
    application = App(only=["firefox"])
    # Merge the base shortcuts with Firefox-specific ones, and add Emacs nav!
    remap = BaseBrowser.remap | EMACS_NAV | {
        "Super-n": "Ctrl-n"
    }
```

## Keybinding Naming Convention

To maintain consistency and ensure the conflict detector works perfectly, please follow this naming convention in `config.py`:

- **Modifiers:** Upper-case first letter (e.g., `Ctrl`, `Alt`, `Shift`, `Super`, `Alt_R`)
  * **Note:** Always use `Alt` instead of `M` (Meta) to avoid confusion. In xremap, `M` is just an alias for `Alt` and does not represent the `Super` key.
  * **Note:** Always use `Ctrl` instead of `C`.
- **Keys:** All lower-case (e.g., `a`, `left`, `home`, `backspace`, `comma`, `f4`)
- **Separator:** Hyphen (`-`)

*Example:* `Ctrl-Shift-left`, `Super-comma`
